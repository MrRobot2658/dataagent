# Quasar：当数据平台学会「理解」业务

> 语义层融合，是 AI Agent 进入企业数据的最后一道门槛

---

最近和一个做 DTC 电商的朋友聊天，他说了一句话让我印象很深：

**「我每天能看到几十个数据指标，但要花两个小时才能知道到底发生了什么。」**

退款率从 3% 跳到 8%。数据平台告诉他「上周退款率 8.2%，比前四周均值 2.7% 高了 3 倍」。

然后呢？没有然后。

他得自己去翻订单明细、找客服聊天记录、问仓库质检、翻供应商档案……才能拼出一个完整的答案：原来是供应商那个批次的皮草色泽不均。

**数据平台只做了第一公里——告诉你「发生了什么」。剩下的九公里，全靠人来走。**

---

## 一、问题出在哪？

大多数数据平台（包括传统的 CDP、BI 工具）只有一层：**SQL 层**。

你把数据灌进去，它给你一个查询界面。你问什么，它查什么。它不知道「退款率」是什么意思、不知道 8% 是正常还是异常、不知道退款和批次质量之间有没有关系。

它是一台**精确但冰冷的机器**。

AI Agent 出现后，很多人以为问题解决了——用自然语言代替 SQL 查询不就好了？

但很快发现：**Agent 能写 SQL，但依然不懂业务。**

Agent 可以精准地输出「退款率 8.2%，SKU FUR-2026-WINTER 占 78%」，但你说「给我分析一下」，它就束手无策了——因为它不知道 SKU 背后有一条供应链，不知道批次里有质检报告，不知道退款率突然升高意味着什么。

**缺少的，是语义层。**

---

## 二、什么是语义层？

语义层是介于原始数据和上层应用之间的**翻译层**。

它做三件事：

**① 给冷数据穿上暖衣。** 数据库里的 `status=refunded` 翻译成业务语言就是「客户退款了，退款原因是质量问题还是尺寸不符？」

**② 把碎片拼成全景。** 把订单表、退款表、质检报告、供应商档案、客服对话……这些散落在不同系统的信息拼成一条完整的业务故事线。

**③ 让 AI 看懂上下文。** Agent 不只是拿到一个数字，而是拿到一个「业务上下文包」——数字 + 定义 + 关联知识 + 异常判断。

过去三年，大家花了很多精力在「怎么让 AI 写 SQL」上。但真正让 AI 能回答业务问题的，不是写 SQL 的能力，而是**理解业务的能力**。

这需要把数据库（结构化数据）和知识库（非结构化知识）在语义层融合起来。

---

## 三、Quasar 的语义融合架构

Quasar 是我们正在构建的下一代数据平台。它的核心差异点，不是又一个 CDP 或 BI 工具，而是**一个内建的语义融合层**。

这个融合层由三部分组成：

### 3.1 Entity Hub（实体中心）

把同一业务实体（用户、订单、商品、供应商、批次）在数据库和知识库中的存在形式统一起来。

数据库里 `user_id = 12345` 是一行记录。知识库里同一个用户是一份画像文档、几条客服对话摘要。

Entity Hub 把它们联起来——**Agent 查一个实体，同时拿到它的精确数据和业务上下文。**

### 3.2 Property Federation（属性联邦）

数据库的字段和知识库的属性做声明式绑定。

`user.total_orders` 不仅是一个数字，它还关联着口径说明文档（「总下单数 = 所有状态为 completed 的订单数量」）、历史趋势、和行业基准。

**字段不再是冰冷的列名，而是有业务生命的概念。**

### 3.3 Relationship Cross-linking（关系跨链）

数据库的外键关系和知识库的实体关系合并为一张统一的关系图。

数据库知道「用户 ➔ 订单」是一对多关系。知识库知道「用户 ➔ 物流投诉」是业务关联。

合并后，Agent 可以走一条完整的链路：

**用户 ➔ 最近订单 ➔ 退款记录 ➔ 退款 SKU ➔ 该 SKU 的质检报告 ➔ 该批次的生产记录 ➔ 供应商的近期质量趋势**

这就是一条完整的**归因链**。

## 四、一个真实的例子

回到开头的故事。

有了语义融合层后，同一个问题：「上周退款率为什么突然升高了？」

**Agent 的思考过程不再只是「查一下数据库」，而是：**

1. **意图理解**：这是一个混合型问题，需要定量（退款率数据）+ 定性（原因分析）

2. **实体解析**：「退款率」➔ Entity Hub 定位到 `orders.status = refunded` 关联的数据集 + 行业基准知识库文档

3. **数据采集（DB 侧）：**
   - 上周退款率 8.2%（前 4 周均值 2.7%），3 倍飙升
   - 78% 的退款来自同一个 SKU：FUR-2026-WINTER
   - 该 SKU 的退款原因分布：58% 质量问题、22% 尺寸不符
   - 质量问题退款集中在 B202603 批次

4. **知识关联（KB 侧）：**
   - B202603 批次的质检报告：**色泽不均，质检退回率 15%**
   - 供应商 A 的近期档案：**投诉率同比上升 300%**
   - 行业基准：8.2% 远超 DTC 皮草行业 3-5% 的正常范围

5. **归因链构建：**
   - 供应商 A 的 B202603 批次存在色泽不均
   - 该批次集中在 FUR-2026-WINTER 这个 SKU
   - 上周这批货大量交付给新客
   - 新客收到后发起退款 ➔ 退款率飙升

**Agent 最终输出的不是数据摘录，而是一份完整的业务分析：**

> 「上周退款率 8.2%，是正常水平的 3 倍。核心原因是 SKU FUR-2026-WINTER，这一个 SKU 占了全部退款的 78%。根本原因是供应商 A 的 B202603 批次存在色泽不均。受影响最大的是近两周下单的新客（80% 退款来自新客）。**
>
> **建议：①立即停售该批次 ②主动联系已购用户补偿 ③要求供应商整改 ④同步开发备选供应商」**

从「数据平台告诉我数字」到「Agent 告诉我发生了什么、为什么、怎么办」——这就是语义层融合带来的质变。

## 五、这不是一个功能，是架构范式

传统数据平台做的是「查询」。你问什么，它回答什么。

Quasar 做的是「理解」。你表达意图，它理解意图、匹配口径、关联上下文、构建归因、生成建议。

这背后不是加一个「智能分析」模块那么简单，而是整个数据平台架构的范式转换：

| 维度 | 传统数据平台 | 语义融合平台 |
|------|-------------|-------------|
| 核心模型 | 表 + 字段 | 实体 + 概念 + 关系 |
| 查询方式 | SQL / NL → SQL | 意图 → 语义表达式 → 多源路由 |
| 知识来源 | 仅有数据库 | 数据库 + 知识库双源 |
| 回答质量 | 精确但孤立 | 精确 + 有上下文 + 可归因 |
| 用户角色 | 分析师（会写 SQL） | 业务人员（说人话） |

---

## 六、为什么是现在？

三个趋势在同时发生：

**第一，AI Agent 的能力已经足够。** 大模型的理解能力让 NL → 语义表达式的翻译准确率达到可用水平。2024 年做不到的事，2025 年可以了。

**第二，企业对数据的要求在升级。** 不是「给我一个仪表盘」，而是「帮我分析一下」。所有人在业务决策上都需要「数据 + 解释」，而不是「数据」。

**第三，基础设施已经成熟。** 向量数据库、知识图谱、语义检索这些组件已经稳定。难的不再是技术选型，而是**架构设计**——怎么把这些组件有机地组合起来。

Quasar 就是在回答这个问题：**当数据库学会理解业务，知识库学会关联数据，Agent 才能真正成为业务伙伴，而不是一个会写 SQL 的高级搜索引擎。**
